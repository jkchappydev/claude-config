## 코드 주석 스타일

**코드에 주석을 달아달라고 요청받으면 아래 스타일을 쓴다.
**회사 프로젝트에는 이 스타일을 적용하지 않는다** — 각 회사 프로젝트의 기존 코드 컨벤션을 따를 것.

- 한 줄 요약으로, 명사형으로 끝낸다 ("~함", "~한다" 같은 종결 어미 없이). 예: "DB 연결 인스턴스 import", "공지 전체를 최신순으로 조회"
- 어려운 용어를 무조건 피하지는 않는다 — "싱글톤", "인스턴스" 같은 핵심 용어는 살짝 써도 되지만 과하게 기술적으로 풀어쓰지 않는다.
- 코드 블록/로직 단위마다 그 위에 한 줄 주석을 붙인다 (라인마다 전부 달 필요는 없음)


## 커밋 규칙 (모든 프로젝트 공통)
- 제목 한 줄 위주로 간결하게. 본문은 꼭 필요할 때만 작성
- Co-Authored-By 등 공동작성자 표기 넣지 않음
- **커밋 전 반드시 사용자에게 커밋 메시지 사전 검사를 받을 것**
- 프로젝트별 CLAUDE.md에 별도 커밋 규칙이 있으면 그쪽이 우선
- **커밋할 때마다 `PROGRESS.md`와 오늘 날짜 워크로그를 함께 업데이트할 것** (아래 "PROGRESS.md / 워크로그 관리" 참고)

## PROGRESS.md / 워크로그 관리 (모든 프로젝트 공통)

컴퓨터 간 작업 인계 + 그날그날 업무일지, 두 목적을 파일 역할을 나눠서 처리한다. 매 커밋 시 두 파일 모두 갱신한다.

### `PROGRESS.md` (프로젝트 루트) — 인계용 스냅샷
- 날짜를 남기지 않고, 항상 "지금 상태"만 담는 짧은 문서로 유지한다. A 컴퓨터에서 B 컴퓨터로 넘어갈 때 이것만 보면 바로 파악되는 게 목적
- 구성은 아래로 고정한다
  1. 최상단에 `# 진행상황`과 문서 목적을 설명하는 인용구(`>`) 유지
  2. `## 완료된 작업` — 끝난 작업을 한 줄(또는 하위 불릿)로 누적. 더 이상 참고할 필요 없는 항목은 정리 시점에 지운다
  3. `## 다음 진행 예정` — 아직 안 끝났거나 다음에 이어서 할 작업. 어디까지 했는지, 왜 그런 결정을 했는지(논의 결과·전환 이유 등)까지 하위 불릿으로 남긴다
- 커밋마다: 이번 커밋으로 완료된 항목은 "완료된 작업"으로 옮기고, 새로 생긴 다음 할 일은 "다음 진행 예정"에 반영한다

### `docs/worklog/YYYY-MM-DD.md` — 날짜별 업무일지
- 프로젝트 루트에 `docs/worklog/` 폴더가 없으면 새로 만든다
- 오늘 날짜 파일(`docs/worklog/2026-07-16.md` 형식)이 없으면 새로 생성하고, 있으면 이어서 추가한다 — 파일을 날짜별로 나눠서 하나가 길어지지 않게 한다
- 커밋할 때마다 오늘 파일에 이번 작업 내용을 추가한다: 무엇을 했는지, 왜 그렇게 했는지(논의·결정 배경), 관련 파일 경로
- 지난 날짜 파일은 수정하지 않는다 — 그날 기록은 그날 파일에 고정
