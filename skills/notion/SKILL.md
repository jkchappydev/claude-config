# Notion 작성 가이드

## 1. 작성 정책

- Notion 작성/수정은 **사용자가 명시적으로 요청했을 때만** 수행한다. 다른 작업의 부수 효과로 자동으로 쓰지 않는다.
- **요청받은 내용만 정확히 반영한다.** 요청하지 않은 섹션·형식을 끼워 넣지 않는다.

## 2. 페이지 관리 (A/B 페이지)

기술 문서는 A/B 두 페이지로 관리한다.

- **B 페이지 (내부용)**: 실제 IP/포트/서비스명으로 작성한다. 내용 추가·수정은 항상 B에 먼저 한다.
- **A 페이지 (외부 공유용)**: B에서 민감 정보를 치환한 버전이다. "A 페이지 동기화해줘" 요청 시에만 업데이트한다.

페이지 ID와 보안 치환 맵은 `page-map.md`를 참고한다. 새 문서가 생기면 `page-map.md`에 추가한다.

## 3. 공통 작성 규칙

모든 문서에 공통으로 적용한다.

### 3.1 문체

- 기술 용어는 **영어 그대로** 쓴다. 설명만 한국어로 작성한다.
- IP·포트·서비스명은 `page-map.md`의 **보안 치환 맵**을 적용한다.
- **현재 상태 기준**으로 서술한다. 검토했던 대안이나 변경 이력은 나열하지 않는다.
- 문장 종결
  - 서술형 문서(기술 문서·프로젝트 개요): "다."로 끝낸다. (예: "채택했다.", "설치한다.")
  - 진행상황 정리: 동사형으로 끝낸다. (예: "설정 완료", "버그 수정")

### 3.2 Callout

콜아웃은 반드시 `<callout>` 태그로 작성한다. `>` 인용구 문법은 쓰지 않는다.

- `>` 문법을 쓰면 아이콘·배경색 없는 plain quote로 쪼개진다. 진짜 콜아웃은 `<callout>` 태그로만 만들어진다.
- **기본 형식** (태그 안 각 줄은 탭으로 들여쓴다. `\t`는 실제 탭 문자):
  ```
  <callout icon="⚠️" color="gray_bg">
  \t**주의**
  \tDB 접근이나 비밀키는 반드시 서버 컴포넌트 또는 `route.ts` 안에서만 다룬다.
  </callout>
  ```
- 코드 블록·불릿 목록 등 여러 블록은 `<callout>`와 `</callout>` 사이에 넣는다.
- 콜아웃 안 코드는 일반 펜스를 그대로 쓴다. 펜스 앞에 `>`를 붙이면 렌더링이 깨진다.

이모지 기준:

- `⚠️` : 실수하기 쉬운 경고, 테스트 대상 안내
- `📌` : 용어 설명, 개념 보충
- `✅` : 확인 포인트, 검증 방법
- `🗂️` : 참고 사례/실제 예시 (본문과 시각적으로 분리)

### 3.3 목록 안에 코드·callout 넣기

- 번호 목록 항목 사이에 code block·callout을 **최상위 블록으로 끼워 넣지 않는다.** 번호가 `1.`로 리셋된다.
- 항목에 딸린 code block·callout은 **그 항목의 자식으로 탭(`\t`) 들여쓰기**해서 넣는다.
  ````
  1. 스키마 정의
  \t```javascript
  \tmodel Notice { ... }
  \t```
  \t<callout icon="📌" color="gray_bg">
  \t\t**설명**
  \t\t내용
  \t</callout>
  2. 다음 단계
  ````
  (콜아웃 안쪽 내용은 자식의 자식이라 탭 두 개.)
- 번호는 `1.` 반복 없이 **실제 순서대로**(`1. 2. 3. ...`) 명시한다.

### 3.4 실행 명령어 기록

- **실제로 실행한 명령어는 빠짐없이** 코드 블록으로 남긴다. 설명만 하고 생략하지 않는다.
- **실행하지 않은 예시성·일반 명령어는 지어내지 않는다.** 값을 모르면 비워두거나 "실행 시 값 확인 필요"로 표시한다.
- 코드 블록은 언어를 명시하고(`bash`, `powershell` 등), 위에 한 줄 설명을 붙인다.
- 실패했거나 버린 명령어는 제외하고, 최종적으로 유효했던 것만 남긴다.

### 3.5 날짜 표기

- 본문·표·callout에 결정 날짜를 **인라인으로 넣지 않는다.** (예: "~로 채택했다 (2026-07-08 결정)." ❌)
- 날짜는 **진행상황 정리의 `📅 날짜 (YYYY-MM-DD)` 헤더에만** 넣는다.

## 4. 문서 유형별 가이드

### 4.1 진행상황 정리

작업 완료·진행중·이슈를 기록한다. **사용자가 "진행상황 정리"를 명시적으로 요청했을 때만** 작성한다.

페이지 구조:

```
📅 날짜 (YYYY-MM-DD)
━━━━━━━━━━━━━━━━━━━━

## ✅ 완료된 작업
- [작업명]: [간략한 설명]

## 🔄 진행 중인 작업
- [작업명]: [현재 상태]

## ❌ 이슈 / 블로커
- [이슈명]: [문제 내용 및 조치 사항]

## 📝 메모
- [기타 참고사항]
```

작성 규칙:

- 한 항목은 **한 줄**로 끝낸다.
- **동사형**으로 작성한다. (문체는 3.1 참고)
- **날짜를 항상 포함**한다.
- **블로커는 즉시 기록**한다. 막힌 이유와 다음 액션을 함께 적는다.
- **링크를 첨부**한다. PR·이슈·커밋 링크를 함께 기록한다.

이모지 기준: ✅ 완료, 🔄 진행중, ❌ 이슈, 🔍 조사중, 🚀 배포

### 4.2 기술 문서

툴 채택 근거, 설치/설정 가이드, 아키텍처 결정 등을 작성한다.

문서 구조:

1. **문서 제목**: `[툴/기술명] 채택 근거 및 설정 가이드`
2. **채택 근거** (항상 첫 번째)
   - `📌 [툴명]이란?` 콜아웃으로 도구 소개를 먼저 작성한다.
   - 채택 이유는 **비교 대상과 함께** 서술한다. (예: "JMeter는 ~지만 k6는 ~라서 채택했다.")
3. **설치/설정 가이드**
   - 번호 매긴 순서로 작성하고, 각 단계에 명확한 소제목을 붙인다.
   - 코드 블록은 언어를 명시하고, 위에 무엇을 하는 코드인지 한 줄 설명을 넣는다.
4. **흐름 요약** (항상 마지막): `A 실행 → B 저장 → C 시각화`

작성 원칙 (3.1 문체에 더해):

- **"왜"를 항상 포함한다.** 설정 하나하나에 이유를 단다.
- **다음 액션을 명시한다.** 문서 마지막에 번호 매긴 다음 단계를 남긴다.

### 4.3 프로젝트 개요

프로젝트 기술스택·인프라 구성을 정리한다.

문서 구조:

1. **문서 제목**: `[프로젝트명] 개요`
2. **기술스택**
   - 언어/프레임워크, 데이터베이스, 인증 방식 등 카테고리별로 나눈다.
   - 버전 정보가 있으면 함께 표기한다. (예: `Java 17`, `Spring Boot 3.3.5`)
   - 선택 이유가 있으면 한 줄로 덧붙인다.
3. **인프라 세팅**
   - 서버 환경(개발/스테이징/운영), 배포 방식, 도커/CI 구성을 정리한다.
   - 배포 프로세스는 흐름으로 요약한다. (예: `빌드 → 포트 정리 → 앱 재시작`)

작성 원칙 (3.1 문체에 더해):

- **카테고리별로 묶어** 표 또는 불릿으로 나열한다. 서술형 문단보다 스캔하기 쉽게 쓴다.
